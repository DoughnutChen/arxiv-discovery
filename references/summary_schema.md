# 摘要输出结构

最终结果由 `scripts/generate_summary.py` 写入 Markdown 文件，默认路径为完整流程输出目录下的 `report.md`。语言使用中文，论文题目保留英文原文。

自动摘要生成支持 OpenAI Responses API 和 OpenAI-compatible chat completions 接口。provider、模型、base URL 和 API key 环境变量名由命令行参数、`.env` / 环境变量、`.local/provider.json` 或内置默认值解析；`configure_provider.py` 可一次性生成本地配置。不能把真实 API key 写入代码、提示词、日志或可提交文件。

## 报告头部

```markdown
# arXiv Discovery：<query>

- 搜索关键词：`<query>`
- arXiv 分类：`<category 或 未指定>`
- 排序方式：`<sort_by>/<sort_order>`
- 处理论文数：`<n>`
- 备注：`<下载失败、解析失败或结果不足等情况>`
```

## 整体摘要

输出一段 200-300 字的整体摘要。语言风格应与单篇摘要一致，概括这一批论文共同关注的问题、主要方法取向和对当前关键词的研究价值。

## 1. English Paper Title

- arXiv ID：`xxxx.xxxxx`
- 链接：[Abstract](https://arxiv.org/abs/xxxx.xxxxx)，[PDF](https://arxiv.org/pdf/xxxx.xxxxx.pdf)
- 作者：A；B；C
- 发布/更新：YYYY-MM-DD / YYYY-MM-DD

### 摘要

**研究问题**

用一段话说明论文要解决的核心问题，重点回答：**为什么研究这个问题、研究对象是什么、核心研究目标是什么**。不拆分为项目符号。

**研究方法**

用一段话说明论文采用的方法，包括：数据来源、理论框架、实证方法或模型设计等，强调**如何解决研究问题**。

**研究结论**

用条目化方式输出 2–4 条核心发现，每条以①②③标号，要求：

① 结论必须是可独立理解的“研究发现单元”，避免背景描述  
② 强调机制、关系或因果结果，而非泛泛描述  
③ 保持信息密度高、表达直接  

### 具体领域
用 1–2 个短语标注论文所属细分方向，例如：
- 自然语言处理 / 机器学习 / AI 安全 / 医疗 AI / 数据治理 / 行为金融 等

### 推荐原因
用 1–2 句说明论文的学术或应用价值，重点强调：
- 对哪个研究方向有启发  
- 或对哪类模型 / 理论 / 实证设计有参考意义  

每篇论文都必须输出 `### 摘要`、`### 具体领域`、`### 推荐原因` 三个小节；`### 摘要` 中的 `**研究问题**`、`**研究方法**`、`**研究结论**` 必须分别单独成行，标题下一行再写正文。研究结论的每个编号结论都必须单独成行。

## 风格样例

```markdown
### 摘要

**研究问题**

本文关注开放银行与数字经济发展背景下，信息技术如何推动产业组织结构重构，重点识别影响新型产业设计形成的关键技术要素，并分析其对市场边界、价值创造方式及治理机制的影响。

**研究方法**

作者采用系统性文献综述方法，对开放银行、平台生态、数字基础设施及产业治理等领域的研究进行归纳整合，从跨学科视角提炼数字经济产业演化中的核心技术驱动机制。

**研究结论**

① 数据开放共享是产业协同的基础条件，开放式接口机制降低了信息不对称，提高了资源匹配效率。

② 数字身份认证与信任机制是数字交易运行的关键制度基础，其核心作用在于降低交易摩擦和提升协作可信度。

③ 平台互联与去中心化治理推动产业组织方式重构，市场结构由封闭式竞争逐步转向开放式生态协同。

### 具体领域

自然语言处理 / 检索增强生成 / AI 系统

### 推荐原因

适合作为数字经济产业重构与平台治理方向的基础理论文献。
```

## 阅读顺序

报告结尾可用 3-5 条中文说明推荐阅读顺序。排序依据优先考虑：与关键词的直接相关性、理论或方法代表性、对研究问题构建的帮助。

## 质量要求

- 不要编造题目、作者、实验结果、理论贡献或论文结论。
- 如果正文解析失败，应明确说明该摘要仅基于题目和 arXiv 摘要。
- 如果论文与关键词只是弱相关，应在“推荐原因”或相关性说明中直接指出。
- 专有名词可以保留英文，并在必要时用中文解释。
